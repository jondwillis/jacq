# Rust Tooling Patterns — Reference for jacq

Technical deep dives into how the major Rust-based developer tools are architected,
with specific patterns jacq should adopt, watch, or avoid.

---

## oxc (oxc-project/oxc)

The JavaScript "Oxidation Compiler" — parser, linter, formatter, transformer, minifier.

### Architecture

~40 crates in `crates/`, organized in layers:
- **Foundation:** `oxc_allocator`, `oxc_ast`, `oxc_span`, `oxc_syntax`, `oxc_diagnostics`, `oxc_index`
- **Processing:** `oxc_parser`, `oxc_semantic`, `oxc_transformer`, `oxc_minifier`, `oxc_codegen`
- **Tools:** `oxc_linter`, `oxc_formatter`, `oxc_language_server`
- **Apps:** `apps/oxlint`, `apps/oxfmt` (separate binaries with both `bin` and `lib` targets)

Workspace resolver v3. Shared deps centralized in root `Cargo.toml`. Single version across all crates. Release profile: `lto = "fat"`, `codegen-units = 1`.

### Key Patterns

**Arena-allocated AST.** `oxc_allocator` (built on `bumpalo`) provides arena-aware `Box<'a, T>` and `Vec<'a, T>` with the arena's lifetime. All AST nodes allocated in a single arena — dropping the arena drops the entire AST in one shot. Key insight: all AST types are 8-byte aligned (they all contain a Vec, Box, Atom, or &str), so alignment calculation is eliminated entirely.

→ *jacq relevance:* Not needed for v0.1 (our IR is small), but if we ever process large plugin directories with many files, arena allocation prevents per-node allocation overhead.

**Typed index wrappers.** `oxc_index` provides newtype wrappers around `u32`/`usize` for safe lookups into symbol tables and scope trees. Prevents mixing up different index types at compile time.

→ *jacq relevance:* If we add a symbol table or cross-reference system for skills/agents, typed indices prevent "wrong ID" bugs.

**`bpaf` for CLI.** Both oxc and Biome use `bpaf` instead of `clap`. Faster compile times, composable argument parsing, built-in autocomplete.

→ *jacq relevance:* We currently use clap. Consider migrating if clap compile times become a bottleneck. For now, clap is fine.

**Dual lib+bin targets.** App crates have both a `bin` and `lib` target so the CLI logic is reusable from NAPI bindings and tests.

→ *jacq relevance:* We already have this pattern (`src/lib.rs` + `src/main.rs`).

**Error reporting via miette fork.** `oxc_diagnostics` uses `oxc-miette` (fork of `miette` v2.7.0) with multiple output reporters: pretty, JSON, Unix, GitHub Actions, GitLab.

→ *jacq relevance:* We use miette already. The multiple-reporters pattern is worth adopting when we add `--format json` output.

### Testing

`insta` snapshot tests for AST and codegen. Test262/Babel/TypeScript conformance suites. E2E tests that transform 100+ npm libraries and execute them in isolated environments.

## Biome (biomejs/biome)

Unified linter/formatter for JS, CSS, JSON, GraphQL, HTML. Formerly Rome.

### Architecture

130+ crates with `biome_*` naming. Per-language crate families:
- `biome_{lang}_syntax`, `biome_{lang}_parser`, `biome_{lang}_formatter`, `biome_{lang}_analyze`

Core infrastructure: `biome_rowan` (CST), `biome_parser` (event-driven), `biome_formatter` (IR + printer), `biome_analyze` (rule trait + registry), `biome_diagnostics`.

### Key Patterns

**Green/Red CST trees (from Roslyn).** Full-fidelity concrete syntax trees that preserve all whitespace and comments. Green tree is immutable and structurally shared (memory efficient). Red tree is lazily constructed with parent pointers and absolute offsets. Typed AST wrappers autogenerated.

→ *jacq relevance:* Not needed — we're not parsing a programming language. But if we ever need to round-trip skill markdown with perfect fidelity (preserving formatting), this is the pattern.

**"Bogus" node error recovery.** Parser never fails. Broken syntax is wrapped in "Bogus" nodes that consumers can safely skip. This means you always get a complete tree, even from invalid input.

→ *jacq relevance:* Our YAML/frontmatter parsing currently fails on first error. A "bogus skill" pattern would let us report all errors at once instead of one-at-a-time.

**GritQL plugins (Biome 2.0).** Plugins are `.grit` files with pattern-matching rules over the CST. Declarative, no Rust required. Limited to analysis (no autofix from plugins yet).

→ *jacq relevance:* If we add lint rules for plugin quality (e.g., "skill must have a description"), a declarative pattern-matching approach is more user-friendly than requiring Rust.

**Per-language crate families.** Clean scaling pattern: add a new language by creating `biome_{lang}_syntax`, `biome_{lang}_parser`, etc.

→ *jacq relevance:* Analogous to per-target crate families: `jacq_claude_code`, `jacq_opencode`, `jacq_codex`. Worth considering when the emitter code grows beyond a single file per target.

**Formatter IR.** Two-phase: source → `FormatElement` IR (hardline, softline, text, indent tokens) → text output. Printer resolves tokens respecting line length.

→ *jacq relevance:* If we need to generate well-formatted YAML/JSON/Markdown output with configurable indentation, a format IR is cleaner than string concatenation.

## SWC (swc-project/swc)

JavaScript/TypeScript compiler in Rust. Babel's Rust successor.

### Architecture

Three parallel language pipelines (ECMAScript, CSS, HTML), each with `_ast`, `_parser`, `_codegen`, `_transforms` crates. `swc_core` as unified facade.

### Key Patterns

**Three-tier visitor pattern.** `swc_ecma_visit` generates three trait variants:
- `Visit` — `&Node` (immutable, for analysis)
- `VisitMut` — `&mut Node` (in-place transformation)
- `Fold` — `Node -> Node` (ownership transfer, tree reconstruction)

→ *jacq relevance:* If we add IR transformations (e.g., applying fallback strategies by rewriting the IR before emission), the `VisitMut` pattern is the right approach.

**WASM plugin system.** Plugins compile to `wasm32-wasi`, run in sandboxed Wasmer/Wasmtime runtimes. Host serializes entire AST to bytes (via `rkyv` zero-copy or `cbor4ii`), sends to guest, guest transforms and returns. Schema versioned for ABI compatibility.

→ *jacq relevance:* This is the reference implementation for sandboxed plugins. If jacq ever supports community-authored emitters, the SWC pattern (serialize IR → WASM guest transforms → deserialize result) is battle-tested. Heavy-weight for v0.1 but the right long-term architecture.

**`rkyv` zero-copy serialization.** For WASM plugin AST transfer, SWC uses `rkyv` which allows reading serialized data directly from bytes without deserialization. Dramatically faster than serde for large ASTs.

→ *jacq relevance:* Not needed now. Would matter if jacq becomes a build daemon processing many plugins concurrently.

## Ruff (astral-sh/ruff)

Python linter/formatter in Rust. 800+ rules, Black-compatible formatting.

### Architecture

50+ crates in flat `crates/`. Entry points: `ruff` (CLI), `ruff_server` (LSP), `ruff_wasm`. Analysis engines: `ruff_linter`, `ruff_python_formatter`. Type checker: `ty_python_semantic` (Salsa-based).

### Key Patterns

**Rule organization — the gold standard.**
```
codes.rs       → maps string codes to (RuleGroup, Rule) tuples
Rule enum      → 1000+ variants (E501, F841, ...)
RuleGroup      → Stable | Preview | Deprecated | Removed
rules/{family}/ → one directory per linter family (pyflakes/, pycodestyle/, ...)
```
`declare_lint_rule!` macro for registration. `Checker` struct is a single-pass AST visitor that builds semantic model while invoking rules.

→ *jacq relevance:* If we add validation rules (e.g., "skill must have description", "hook timeout should be positive", "name must be kebab-case"), this registry pattern scales cleanly. The `RuleGroup` concept (Stable/Preview/Deprecated) maps well to capability maturity levels.

**Fix applicability levels.** Three tiers:
- **Safe** — semantics-preserving, auto-applied
- **Unsafe** — behavior-changing, requires `--unsafe-fixes`
- **DisplayOnly** — suggestion only, no auto-apply

→ *jacq relevance:* Excellent UX pattern. When jacq's analyzer suggests fallback strategies, they could have applicability: "safe" (skip a non-critical feature), "unsafe" (convert hook to instruction — changes behavior), "display" (inform only).

**`insta` snapshot testing workflow.** Fixture files in `resources/test/fixtures/`. `cargo test` generates snapshots. `cargo insta review` to accept/reject. `.snap` files committed alongside code.

→ *jacq relevance:* We added insta as a dev dependency but haven't used it yet. For Phase 6 (dogfooding), snapshot tests on emitted output would catch regressions in generated files. Ruff's workflow is the template to follow.

**Salsa for incremental computation.** The type checker (`ty_python_semantic`) uses Salsa's query-based memoization — compute on demand, cache results, invalidate when inputs change.

→ *jacq relevance:* If jacq becomes a watch-mode build tool (rebuild on file change), Salsa's incremental query framework is the right foundation. Overkill for v0.1 batch mode.

## Deno (denoland/deno)

JavaScript/TypeScript runtime with built-in toolchain.

### Key Patterns

**`#[op2]` proc macro.** Exposes Rust functions to JS by generating all glue code from a plain function signature. Type conversions, error handling, fast/slow dispatch paths generated automatically.

→ *jacq relevance:* If jacq exposes a Rust API to WASM/JS plugins, a proc macro that generates the binding code from a plain Rust function is the right pattern.

**`extension!` macro for composable bundles.**
```rust
deno_core::extension!(
    my_ext,
    deps = [deno_console],
    ops = [op_my_function],
    esm = ["js/my_module.js"],
    state = |state| { state.put(MyState::new()); }
);
```
Bundles ops, JS sources, state initialization, and dependencies into a composable unit.

→ *jacq relevance:* If we modularize emitters as separate crates, a similar macro could bundle an emitter's target definition, capability matrix, templates, and validation logic into a single registration.

**`OpState` type-erased state bag.** Simple pattern for passing arbitrary state through a plugin system without generic parameter explosion. Uses `GothamState` (type-keyed `HashMap<TypeId, Box<dyn Any>>`).

→ *jacq relevance:* Useful if emitters need shared state (e.g., template cache, computed analysis results) without threading generics everywhere.

## Summary: What jacq Should Adopt Now vs. Later

### Adopt Now (v0.1)
| Pattern | From | Why |
|---------|------|-----|
| `insta` snapshot tests on emitted output | Ruff | Catch regressions in generated files |
| Centralized workspace deps in root `Cargo.toml` | All five | Already doing this |
| Dual lib+bin crate layout | oxc | Already doing this |
| `miette` for error reporting | oxc | Already doing this |

### Adopt Soon (v0.2-0.3)
| Pattern | From | Why |
|---------|------|-----|
| Per-target crate families | Biome | When emitter code grows beyond single files |
| Rule registry with codes + groups | Ruff | When we add validation rules |
| Fix applicability levels | Ruff | Better UX for fallback suggestions |
| Multiple output reporters (JSON, GitHub Actions) | oxc | For CI integration |

### Adopt Later (v1.0+)
| Pattern | From | Why |
|---------|------|-----|
| WASM plugin system for community emitters | SWC | When third parties need to add targets |
| Arena-allocated IR | oxc | When processing large plugin directories |
| Salsa incremental computation | Ruff | When adding watch-mode builds |
| Extension macro for modular emitters | Deno | When emitters become separate crates |
| `bpaf` CLI parser | oxc, Biome | If clap compile times become a problem |
